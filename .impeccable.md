# Hive OS Design Context

## Design Context

### Users
Hive OS is for a small multi-user human + AI-agent team running on a private workstation/server. Users include an environment admin (`kuya` in the laptop prototype) and regular agent users/collaborators (`aris` now; future team members later). They need to chat with agents, manage private/shared projects, invite/remove collaborators, inspect project wiki/tasks/artifacts, and eventually choose runner backends.

### Brand Personality
Practical, calm, operator-grade, and privacy-aware. The interface should feel like a real production agent console rather than a decorative SaaS mockup.

### Aesthetic Direction
Do not invent a new style. Hive OS should visually stay close to Hermes Desktop on desktop: same light chrome, side panes, compact rows, chat-first layout, subtle blue accent, codicon-style utility controls, and tool/session density. On mobile, keep the same visual tokens but adapt layout to a clean ChatGPT-mobile-like experience: full-screen chat, slide-over/menu drawer, simple top bar, bottom composer, minimal chrome.

### Anti-reference
Do not make it look like a generic blue-purple AI SaaS dashboard, and do not redesign Hermes Desktop styling. The task is responsive adaptation plus added Hive OS menus, not a brand restyle.

### Design Principles
1. Desktop parity first: keep Hermes Desktop layout/style patterns as the reference.
2. Mobile adaptation, not redesign: collapse panes into drawers/sheets while preserving tokens and component feel.
3. Privacy visibility: project membership/access state must be clear without making admin users feel omnipotent.
4. Chat stays primary: projects/wiki/tasks/runners are navigation contexts around chat, not a separate heavy dashboard.
5. Runner-agnostic wording: use “Runners” and “Agents”; Hermes is one runner, not the whole product.
